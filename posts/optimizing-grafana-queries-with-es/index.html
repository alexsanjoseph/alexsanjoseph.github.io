Optimizing Grafana Queries With Elasticsearch

Created: 2021 Feb 24 , Updated: 2021 Mar 7
elasticsearch grafana optimization
8 min read

Intro

Grafana, a visualization framework, was originally created a single developer because he felt that he could improve Kibana, a vizualization framework which comes bundled with Elasticsearch. Over time, it was co-opted by other ecosystems like Prometheus, Influx as their own standard way of visualization as well, allowing grafana to become one of the industry standards in the field of observability and visualization of metrics.

At VIMANA, we use grafana dashboards extensively for monitoring. This includes both the operations data (about the infrastructure), as well as the device data (from the sensors). The data in the backend is hosted in Elasticsearch, and Grafana makes Elasticsearch queries to render the visualizations

Grafana dashboards are very inuititive to build (except the landing page - I can never make sense of it!). However, we have observed over time that single dashboards which could have only panels in the low 10s can end up causing a huge load on a reasonably large Elasticsearch cluster. Even a small number of users simulataneously querying Elasticsearch with one of these dashboards can cause ES CPU usage to spike causing lack of availability and reducing indexing speeds cluttering up the pipeline. This document explains a few things we have learned while trying to optimize Grafana dashboards.

How to profile your Grafana query

Panel Inspector

Grafana comes packaged with with a Panel Inspector Tool that allows you to:

  • Show the request being made to the server
  • Show the response from the server
  • Get statistics of how much time the query took

This is a good place to start your profiling as it tells us what’s the query that is being sent to Elasticsearch. In many cases, just getting an understanding of the actual query that’s being made helps us understand where the bottlenecks are.

The panel inspector allows you to debug individual panels in grafana quite well. However, in cases where there are dashboards with many panels, you might be interested in knowing the interactions between the queries the different panels make and how the multiple requests are sent from the dashboard. In this case, we have found that it is best to use the developer tools from the browser to get a holistic view of the dashboard.

Using Browser Devtools

Browser usually have a Developer Tools option that can help in all sorts of debugging. Developer tools have a lot of information, but in this specific case, we’re interested in the Network tab, which shows what are the queries that are made by the browser to the backend.

Grafana queries to ES are sent to the _msearch end point you can filter by that prefix to see all the queries made by the single dashboard load. For each query, we can also see the time taken by the request, and also the query details, payload etc can be accessed by going into the details page of individual queries.

Using Logs from Elasticsearch

In you have query logging enabled in ES, we can also see the same queries from the Elasticsearch side of things by going over the query logs. This can be either by logging into the container and looking at the logs, or more likely, by persisting this in a logging system (like an ELK stack) and looking at the logs in bulk. This is also the most efficient way to get summary statistics of the queries. There are two kinds of queries in this case that you might be interested in

  • Dashboard logs for each page loag request from a Grafana dashboard
  • Query logs for each individual query made.

Using Elasticsearch Dev Tools

Once you have identified the queries that you want to profile using the above techniques, we have to figure out how to improve them. Sometimes it might be obvious why the queries are slow, but in other cases, you might want to understand what Elasticsearch is doing in the background for it to take so much time. Kibana has a Dev Tools tab which has a search profiler, which can help you debug how single queries are executed.

Observations and Recommendations

‘Now’ Queries are not Cached

By default, Grafana dashboards are tranched by time into executing the query for the last few hours, minutes, etc. These queries are parsed by Grafana to find the current time and sent to the Elasticsearch as absolute time range queries. However, one issue with the default Now queries is that because the time ranges of each query is different.

Elasticsearch natively implements query caching which can quickly return results that have been executed previously without stressing the cluster. However, the cache is hit only when the queried time range exactly matches a previously queried time. Because of this behavor, the user doesn’t get any benefit of using the Elasticsearch cache while doing very similar repeated queries.

Recommendations

  • Query Using absolute times in the dashboard the the time range you’re looking for is in the past.
  • If relative times are necessary, but use the / notation (now - 1d/d, now - 2h/h) etc to tranch the time range to fixed boundaries

Time aggregations can become expensive

We’ve seen two cases where aggregations based on time where the query can become really expensive.

  • Date histogram with small buckets - This is a common case where the user is trying to zoom into individual data points, since grafana only allows the user to view an aggregation and not necessarily the source data. However, the number of aggregation buckets can grow really large if the bucket size is small. For example, we have had users trying to look at last one day of data with a bucket size of 250ms. This comes to around 86400 * 4 buckets which can overload 1) the cluster, 2) network, 3) Grafana trying to plot all these points.
  • Term aggregations based on time - This is similar to the other case, but where the user tries to work around the situation of having small buckets by keeping each timestamp in individual buckets. While this can be alright if the number of timestamps is low, in many cases the sampling rate of the data is very high and we could end up with millions of data points in a few days time range if we are not careful.

Recommendations

  • Let Grafana manage the bucket size (default behavior)
  • Limit queries across smaller time ranges if you’re looking for very granular data
  • Don’t use terms queries for fields with large cardinality like time ranges

A Single Dashboard can make 100s of queries

In many cases, people making the dashboard likes to have all the information they want in one screen. While this might work for dashboards having less data, with more data/aggregation heavy dashboards, this can mean that each dasboard is sending 100s of queries in one shot while it is getting loaded.

Recommendations

  • Use Group By instead of multiple queries - In many cases, you want to see the behavior or multiple timelines which different by a single field (say comparing multiple IoT devices) at the same time for comparison. In such a case, instead of having an independent query for each filter (Query 1 filtered device_1, Query 2 filtered by device_2), it is better to do a generic filter (filter by device_1 OR device_2…) and then add a Group By option (before the mandatory Then by timestamp).
  • Use different dashboards for each use case rather than putting everything into one dashboard
  • Use Rows to hide set of panels that are not always relevant (rows are loaded lazily only while expanding by grafana)

Queries hitting multiple shards can overload the system

If most of your aggregations are about a single dimension (a data item, a device etc), if the data about that dimension is stored across all the shards (default round robin routing behavior), then each query will be hitting all the nodes of the cluster. While this can potentially increase the querying speed, in many cases, this also invovles bringing in data from all partitions into a single node for aggregation and also taxes all the nodes instead of isolating the load into a single node/shard. This can cause the cluster load to go up when lots of queries are happening at the same time.

Recommendations

  • Route documents while storing the data so that each commonly queried dimention is stored in a single shard/node as much as possible.

Conclusion

The contents of this post have been created after a lot of experimentation and reading the excellent Elastic Documentation. We’ve been able to make an order of magnitude increase in querying efficiency following the above guidelines. However, it is possible that the suggestions don’t work as is for your system and data model! If so please post a comment so that we can learn together!
